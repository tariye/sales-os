# v0.87 Production Bridge Runbook

v0.87 is the external runtime release. It does not add intelligence features. It makes the existing authenticated ledger API deployable as the shared memory bus for ChatGPT Actions or another approved client.

## Production Contract

```text
GitHub
  owns code, docs, tests, migrations, generated memory exports

SQLite/API host
  owns live persistent intelligence

ChatGPT Action or approved client
  owns controlled conversational access
```

Chats should never receive unrestricted SQL or filesystem access. They should only use controlled operations:

- create ledger entry
- search entries
- retrieve context
- get action queue
- get critical alerts
- acknowledge alert
- log feedback
- update decision

## Required Host Properties

- Stable HTTPS hostname, for example `https://api.infoanalyzer.example`
- Persistent disk mounted for the SQLite file
- `INFO_ANALYZER_API_KEY` stored as a host secret, not in GitHub
- `INFO_ANALYZER_DB_PATH` set to the persistent database location
- Automated SQLite backups
- Restart-safe process manager
- One controlled writable SQLite instance

Do not place `info_analyzer.db` on ephemeral disk. A restart must not erase the ledger.

## SQLite Runtime Requirements

The app now applies these pragmas on every connection:

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
```

Verify them through:

```bash
curl -H "Authorization: Bearer $INFO_ANALYZER_API_KEY" \
  https://api.infoanalyzer.example/api/v1/health
```

Expected `data.sqlite` fields:

```json
{
  "journal_mode": "wal",
  "foreign_keys": true,
  "busy_timeout_ms": 5000,
  "quick_check": "ok"
}
```

## Environment

## Backups

Create a backup:

```bash
python3 tools/backup_sqlite.py \
  --db "$INFO_ANALYZER_DB_PATH" \
  --backup-dir /var/backups/info-analyzer \
  --manifest /var/backups/info-analyzer/latest-backup.json
```

Restore only with explicit confirmation:

```bash
python3 tools/restore_sqlite.py \
  --backup /var/backups/info-analyzer/info_analyzer-YYYYMMDD-HHMMSS.db \
  --target "$INFO_ANALYZER_DB_PATH" \
  --confirm RESTORE_INFO_ANALYZER_DB
```

After restore:

```bash
python3 tools/memory_manager.py
```

Then run the cross-chat acceptance test.

## Register the Chat Tool

Import:

```text
docs/openapi/info-analyzer-v1.yaml
```

Configure the Action/App with:

```text
Base URL: https://api.infoanalyzer.example
Authentication: Authorization: Bearer <INFO_ANALYZER_API_KEY>
```

The schema paths include `/api/v1/...`, so the base URL should be the host root, not `/api/v1`.

## Cross-Chat Acceptance Test

Run locally against the deployed HTTPS API:

```bash
python3 tools/cross_chat_acceptance.py \
  --base-url https://api.infoanalyzer.example \
  --api-key "$INFO_ANALYZER_API_KEY" \
  --write-proof docs/proof/v0.87-cross-chat-acceptance.json
```

The script simulates:

- Chat A writing an operator setup memory
- Chat A idempotency replay
- Chat B retrieving the memory
- Chat B retrieving related context
- Chat B acknowledging the alert/action
- Chat C logging feedback
- Invalid-key 401 envelope

## Production Acceptance Criteria

| Test | Required result |
| --- | --- |
| Cross-chat write | Chat A creates a ledger entry |
| Cross-chat retrieval | Chat B retrieves the same entry |
| Idempotency replay | Duplicate request returns the original operation |
| Alert acknowledgement | Alert status changes correctly |
| Feedback loop | Action result updates linked memory |
| Auditability | Source chat and request metadata remain available |
| Persistence | Entry survives server restart |
| Restore | Backup can rebuild the database |
| Security | Invalid or missing key receives stable 401 |
| Secret hygiene | API key never appears in logs, exports, or GitHub |

The bridge is not production-complete until the acceptance test passes against the deployed HTTPS URL, not just localhost.
